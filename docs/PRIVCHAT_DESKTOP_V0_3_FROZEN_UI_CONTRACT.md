# PRIVCHAT_DESKTOP_V0_3_FROZEN_UI_CONTRACT

Status: Frozen (implementation contract)  
Owner: privchat-iced  
Version: v0.3  
Supersedes: v0.2  
Depends on:
1. `PrivChat Desktop (Iced) Architecture Spec v0.1`

## 1. Intent

This document freezes UI-facing contracts for the desktop client:

1. Message names and payload fields.
2. Presentation state struct fields.
3. Runtime sequence rules.
4. Timeline patch enum and apply rules.

Any implementation in `app`, `sdk`, `presentation`, and `ui` MUST conform to this document.

## 2. Canonical Naming Freeze

### 2.1 Identity Mapping from SDK

UI contracts consume `privchat-sdk` types through the sdk bridge. The table below defines the canonical mapping:

| UI Contract Name | SDK Origin | Rust Type | Notes |
|---|---|---|---|
| `channel_id` | `StoredChannel.channel_id` | `u64` | SDK calls it channel, not conversation |
| `channel_type` | `StoredChannel.channel_type` | `i32` | Always paired with `channel_id` |
| `user_id` / `from_uid` | `StoredMessage.from_uid` | `u64` | Sender identity |
| `message_id` | `StoredMessage.message_id` | `u64` | Local message identity (DB primary key) |
| `server_message_id` | `StoredMessage.server_message_id` | `Option<u64>` | Remote-confirmed identity |
| `client_txn_id` | (UI-only, see 2.4) | `u64` | Local-send tracking, not in SDK |
| `open_token` | (UI-only, see 6.1) | `u64` | Subscription guard, not in SDK |
| `timeline_revision` | (UI-only, see 3) | `u64` | Patch-stream watermark, not in SDK |

### 2.2 Naming Rules

1. `server_message_id` is the canonical remote-confirmed message identity. It maps from SDK's `server_message_id: Option<u64>` on `StoredMessage`.
2. `message_id` is the canonical local message identity (SDK's DB primary key).
3. `client_txn_id` is a UI-layer local-send tracking identity, not present in SDK. It is a process-local monotonic counter used only within the Iced application lifetime.
4. `channel_id + channel_type` always appear as a pair in SDK APIs. UI contracts MUST preserve this pairing.
5. SDK canonical identity fields (`channel_id`, `channel_type`, `message_id`, `server_message_id`, `pts`) MUST keep their names and semantics in contracts. UI display fields MAY use presentation names (for example `content -> body`) with explicit mapping notes.

### 2.3 `message_id` vs `client_txn_id` Boundary Rules

1. `message_id` (SDK local DB primary key) MUST NOT be used as retry identity or UI patch anchor for send flows.
2. `client_txn_id` (UI-only) MUST NOT be persisted as canonical SDK identity or passed to SDK storage APIs.
3. Local echo replacement MUST match by `client_txn_id` first, never by `message_id`. See Section 7.3.
4. `message_id` is carried on `MessageVm` for app/presentation bookkeeping (for example tracing back to SDK payloads), but UI widgets MUST use `TimelineItemKey` for item identity, not `message_id`.

### 2.4 UI-Only Identities

These identities exist only in the UI/presentation layer and have no SDK counterpart:

1. `client_txn_id: u64` — process-local monotonic counter, generated by Presentation on `SendPressed`. Used for local echo tracking, send-state updates, and local-to-remote replacement. Does not survive process restart.
2. `open_token: u64` — process-local monotonic counter, generated by Presentation on conversation open/retry. Used for subscription dual guard. See Section 6.1.
3. `timeline_revision: u64` — per-channel patch-stream watermark. See Section 3.

## 3. Revision Semantics Freeze

`timeline_revision` is defined as:

1. Per-channel (scoped to `channel_id + channel_type`).
2. Single monotonic **patch-stream** watermark.
3. Shared across patch-stream events only: live updates, send-state updates, unread marker updates, removals — all delivered via `TimelinePatched`.

**Patch-stream** apply rule:

1. If incoming `TimelinePatched.revision <= state.timeline.revision`, the patch MUST be ignored.
2. If incoming `TimelinePatched.revision > state.timeline.revision`, apply and set watermark to incoming revision.

**Excluded from revision gate:**

1. `HistoryLoaded` does NOT participate in revision gating. It uses `server_message_id`-based deduplication instead. See Section 10.
2. `ConversationOpened` snapshot initializes the watermark; it is not subject to comparison against a prior value.

No split revision domains are allowed within the patch stream in v0.3.

```rust
        channel_id: u64,

    TimelinePatched {
        channel_id: u64,
        channel_type: i32,
        open_token: OpenToken,
        revision: TimelineRevision,
        patch: TimelinePatchVm,
    },
    LoadOlderTriggered {
        channel_id: u64,

    },
    HistoryLoaded {
        channel_id: u64,
        channel_type: i32,

        page: HistoryPageVm,
    },
    HistoryLoadFailed {
        channel_id: u64,

        open_token: OpenToken,

        channel_id: u64,
        channel_type: i32,
        at_bottom: bool,
        first_visible_item: Option<TimelineItemKey>,
    },
}

pub enum TimelineItemKey {
    Local(ClientTxnId),
    Remote { server_message_id: u64 },
}

pub enum MessageSendStateVm {
    Queued,
    Sending,
    Sent,
    FailedRetryable { reason: UiError },
    FailedPermanent { reason: UiError },
    Retrying,
}

/// Mapped from SDK's StoredMessage at the sdk bridge/presentation adapter boundary.
pub struct MessageVm {
    pub key: TimelineItemKey,
    pub channel_id: u64,
    pub channel_type: i32,
    pub message_id: u64,                        // SDK local DB primary key
    pub server_message_id: Option<u64>,          // SDK remote-confirmed id
    pub client_txn_id: Option<ClientTxnId>,      // UI-only, for local echo
    pub from_uid: u64,                           // SDK sender identity
    pub body: String,                            // mapped from StoredMessage.content
    pub message_type: i32,                       // SDK message_type
    pub created_at: i64,                         // SDK created_at
    pub pts: Option<u64>,                        // SDK read progression cursor; None for local-only echoes
    pub send_state: Option<MessageSendStateVm>,  // Some for own messages, None for others
    pub is_own: bool,
    pub is_deleted: bool,
}

/// Mapped from SDK's TimelineSnapshot at the sdk bridge/presentation adapter boundary.
pub struct TimelineSnapshotVm {
    pub revision: TimelineRevision,
    pub items: Vec<MessageVm>,
    pub oldest_server_message_id: Option<u64>,   // pagination cursor, from SDK oldest_message_id
    pub has_more_before: bool,                   // from SDK has_more_before
    pub unread_marker: UnreadMarkerVm,
}

pub struct HistoryPageVm {
    pub items: Vec<MessageVm>,
    pub oldest_server_message_id: Option<u64>,   // pagination cursor
    pub has_more_before: bool,
}

pub struct UnreadMarkerVm {
    pub first_unread_key: Option<TimelineItemKey>,
    pub unread_count: u32,
    pub has_unread_below_viewport: bool,
}

pub enum TimelinePatchVm {
    ReplaceLocalEcho {
        client_txn_id: ClientTxnId,
        remote: MessageVm,
    },
    UpsertRemote {
        remote: MessageVm,
    },
    UpdateSendState {
        client_txn_id: ClientTxnId,
        send_state: MessageSendStateVm,
    },
    RemoveMessage {
        key: TimelineItemKey,
    },
    UpdateUnreadMarker {
        unread_marker: UnreadMarkerVm,
    },
}
```

### 4.1 `MessageVm.send_state` Rules

1. When `is_own == true`, `send_state` MUST be `Some(...)`.
2. When `is_own == false`, `send_state` MUST be `None`.
3. An `UpdateSendState` patch targeting an item where `send_state` is `None` MUST be ignored and logged as contract violation.

### 4.2 Send Failure Surface

1. v0.3 does not define a standalone `SendFailed` `AppMessage`.
2. Send failure is represented by `TimelinePatched` with `TimelinePatchVm::UpdateSendState { .. Failed* .. }`.

### 4.3 `HistoryLoaded` — No Revision Field

`HistoryLoaded` does not carry a `revision` field. History pages are not part of the patch-stream watermark domain. Deduplication and ordering are handled by `server_message_id`. See Section 10.

### 4.4 Pagination Cursor

The pagination cursor is `oldest_server_message_id: Option<u64>`. This maps directly to SDK's `oldest_message_id` from `TimelineSnapshot` and to `before_server_message_id` in `MessageHistoryGetRequest`. It is always a `server_message_id`, never a local `message_id` or `client_txn_id`.

### 4.5 `MessageVm.pts` Rule

1. `pts` is the canonical read progression field from SDK semantics.
2. Current SDK `StoredMessage` does not expose a top-level `pts` field; adapter MUST project `pts` from message metadata payload (for example `extra.pts` or `extra.version`) when present.
3. If projection cannot resolve `pts`, value MUST be `None`.
4. `mark_read` resolution MUST use the latest visible timeline item with `pts.is_some()`.
5. Local-only echoes MAY have `pts == None` and MUST be skipped when computing `last_read_pts`.

## 6. Open/Subscription Lifecycle Freeze

### 6.1 `OpenToken` Generation

`OpenToken` is a **process-local monotonic counter**, incremented and assigned only by Presentation when handling `ConversationSelected` or `RetryOpenConversation`. Uniqueness is required only within the current process lifetime. It MUST NOT be random.

### 6.2 Lifecycle Rules

On `ConversationSelected(channel_id, channel_type)` or `RetryOpenConversation(channel_id, channel_type)`:

1. `Presentation` MUST allocate a new `open_token`.
2. Existing timeline subscription MUST be detached.
3. New timeline subscription MUST be attached for target channel.
4. Async events MUST pass dual guard:
   1. `channel_id == active_chat.channel_id && channel_type == active_chat.channel_type`
   2. `open_token == active_chat.open_token`

Dual-guard applicability:

1. `ConversationOpened`
2. `ConversationOpenFailed`
3. `TimelinePatched`
4. `HistoryLoaded`
5. `HistoryLoadFailed`

If either guard fails, event MUST be ignored.

### 6.3 `RetryOpenConversation` Semantics

1. `RetryOpenConversation` follows the same open pipeline as `ConversationSelected` (allocate token, detach, attach).
2. It does NOT alter `session_list` selection state.
3. UI MAY display a different loading indicator to distinguish retry from initial navigation.

## 7. Runtime Sequence Freeze

### 7.1 Open Conversation

1. `ConversationSelected` or `RetryOpenConversation` MUST allocate a new `open_token` and switch to opening state.
2. Previous subscription MUST be detached before attaching the new one.
3. `ConversationOpened` MUST pass dual guard before apply.
4. Open snapshot apply MUST initialize at least:
   1. `timeline.revision`
   2. `timeline.items`
   3. `timeline.oldest_server_message_id`
   4. `timeline.has_more_before`
   5. `unread_marker`
5. `ConversationOpenFailed` MUST keep selected context and expose retry affordance via `RetryOpenConversation`.

### 7.2 Send + Local Echo

1. `SendPressed` with empty/whitespace draft MUST be ignored.
2. `ClientTxnId` MUST be generated (process-local monotonic counter) before sdk bridge send call.
3. Local echo item MUST be inserted in the same `update()` tick as `SendPressed`.
4. Draft clear MUST happen only after local echo insertion succeeds.
5. Initial send state MUST be `Some(Queued)` or `Some(Sending)`.

### 7.3 Remote Confirmation Replacement

1. Remote confirmation MUST arrive through `TimelinePatched`.
2. Replacement matching order MUST be:
   1. By `client_txn_id` (`ReplaceLocalEcho`)
   2. Else by `server_message_id` (`UpsertRemote`)
3. Successful replacement MUST not leave duplicate local and remote logical items.
4. Replacement MUST preserve item visual continuity and viewport anchor.

### 7.4 Send Failure + Retry

1. Failure is represented only as `TimelinePatched(UpdateSendState { Failed* })`.
2. Failure mapping MUST target exactly one local item by `client_txn_id`.
3. `RetrySendPressed` MUST act on existing local item identity.
4. Retry MUST NOT create second timeline item for the same logical message.

### 7.5 Load Older Pagination

1. `LoadOlderTriggered` MUST be ignored when already loading or no more history.
2. Pagination cursor MUST come from `oldest_server_message_id`, never from local echo or `client_txn_id`.
3. `HistoryLoaded` MUST pass dual guard (`channel_id + channel_type + open_token`). It does NOT pass revision gate.
4. Prepend MUST deduplicate by `server_message_id` against existing timeline items and preserve viewport anchor.
5. Failure MUST reset loading state and keep retry path available.

### 7.6 Incoming, Unread, and `mark_read`

1. At bottom:
   1. New remote items SHOULD keep pinned-to-bottom behavior.
   2. `mark_read` SHOULD be emitted debounced.
2. Not at bottom:
   1. UI MUST NOT auto-jump.
   2. Unread marker MUST update.
   3. `mark_read` MUST NOT advance beyond visible cursor.
3. Returning to bottom MUST clear unread marker and emit one latest `mark_read`.

## 8. Patch Apply Rules Freeze

Common preconditions for all `TimelinePatched` processing:

1. Pass dual guard from Section 6.
2. Pass revision gate from Section 3.

### 8.1 ReplaceLocalEcho

1. Locate local item by `TimelineItemKey::Local(client_txn_id)`.
2. If `remote.server_message_id` is `None`, patch MUST be ignored and logged as contract violation.
3. Replace in-place with `remote` item and force key to `TimelineItemKey::Remote(server_message_id)`.
4. Remove any duplicate remote item with same `server_message_id` if already present.
5. Preserve visual position and viewport continuity.

### 8.2 UpsertRemote

1. If `remote.server_message_id` is `None`, patch MUST be ignored and logged as contract violation.
2. If remote key already exists, update that item in-place.
3. Else append/insert according to timeline order policy.
4. Must not create duplicate `TimelineItemKey::Remote(server_message_id)`.

### 8.3 UpdateSendState

1. Find item by `client_txn_id`.
2. If item's `send_state` is `None`, MUST be ignored and logged as contract violation.
3. Apply monotonic state transition only.
4. Illegal backward transition MUST be ignored and logged.

Allowed transitions:

1. `Queued -> Sending | FailedRetryable | FailedPermanent`
2. `Sending -> Sent | FailedRetryable | FailedPermanent | Retrying`
3. `FailedRetryable -> Retrying`
4. `Retrying -> Sending | FailedRetryable | FailedPermanent | Sent`

Terminal states (no outgoing transitions):

1. `Sent` — delivery confirmed; terminal.
2. `FailedPermanent` — unrecoverable failure; terminal.

Any incoming transition from a terminal state MUST be ignored and logged.

### 8.4 RemoveMessage

1. Locate item by key.
2. Mark as deleted or remove physically based on rendering policy.
3. The policy MUST be consistent per build target and not vary at runtime.

### 8.5 UpdateUnreadMarker

1. Replace `unread_marker` atomically.
2. Must not mutate unrelated timeline items.

## 9. Retry Semantics Freeze

1. Retry is logical-item resurrection, not re-insertion.
2. `RetrySendPressed` MUST target an existing local item by `ClientTxnId`.
3. UI MUST NOT implement retry as delete-old-item then insert-new-item.
4. During retry, same timeline item MUST keep stable visual identity and position.
5. sdk bridge handling MUST preserve logical continuity even if SDK internals requeue.

## 10. Pagination Freeze

1. `LoadOlderTriggered` MUST be ignored when `is_loading_more == true` or `has_more_before == false`.
2. Pagination cursor MUST be derived from `oldest_server_message_id`, never from local echo. This maps to SDK's `before_server_message_id` in `MessageHistoryGetRequest`.
3. `HistoryLoaded` MUST pass dual guard (`channel_id + channel_type + open_token`). It does NOT pass revision gate.
4. `HistoryLoaded` deduplication: incoming items MUST be deduplicated by `server_message_id` against existing timeline items. Duplicates are dropped silently.
5. Prepend ordering: deduplicated items are prepended before the current oldest remote item, preserving their original order.
6. After prepend, `oldest_server_message_id` and `has_more_before` MUST be updated from `HistoryPageVm`.
7. `is_loading_more` MUST be reset on both success and failure paths.
8. Viewport anchor MUST be preserved across prepend.

## 11. Read Marker Freeze

1. `ViewportChanged` is the only UI intent that drives read progression.
2. `ViewportChanged` may fire at high frequency during scroll.
   1. UI-side state updates (`first_visible_item`, `at_bottom`) MAY be applied immediately.
   2. app-side `mark_read` emission MUST be debounced (recommended: 300–500ms trailing edge).
   3. No two `mark_read` calls for the same channel should be in-flight simultaneously.
3. When `at_bottom == true`, `mark_read` SHOULD be emitted by app flow with debounce.
4. When `at_bottom == false`, UI MUST NOT auto-advance read beyond visible cursor.
5. Unread marker updates MUST be applied only through `UpdateUnreadMarker` patch.
6. `mark_read` maps to SDK's `project_channel_read_cursor(channel_id, channel_type, last_read_pts)`.
7. `mark_read` MUST use SDK's canonical read progression field (`pts`) only. UI/presentation MUST NOT substitute `server_message_id` for `pts`. The app + sdk bridge path is responsible for resolving the latest visible timeline item to its corresponding `pts` value before calling the SDK.

## 12. Invariants (Must Hold)

1. No duplicate logical message after local-to-remote replacement.
2. No stale async event can mutate current active chat (dual guard).
3. No equal/older revision can mutate timeline via patch stream (revision gate).
4. Retry preserves the same logical item continuity.
5. History prepend preserves viewport anchor.
6. `send_state` is `Some` if and only if `is_own == true`.
7. `channel_id + channel_type` always appear as a pair in all messages and state.

## 13. Contract Tests (Minimum)

1. Select channel A then immediately B; all late A async messages are ignored by dual guard.
2. Send one message; exactly one local item appears before network response.
3. Success patch replaces local item with remote without duplication.
4. Failed send shows failed state on same item; retry reuses same item.
5. Equal revision replay is ignored and render output is unchanged.
6. Pagination prepend does not jump viewport.
7. `HistoryLoaded` with older `server_message_id` than current live events is still applied (not rejected by revision gate).
8. `HistoryLoaded` deduplicates by `server_message_id` against existing timeline items.
9. `RetryOpenConversation` allocates new `open_token`; old async events with previous token are rejected.
10. `UpdateSendState` targeting an `is_own == false` item is ignored and logged.
11. Transition from `Sent` or `FailedPermanent` is ignored and logged.

## 14. Out of Scope

The following features are explicitly excluded from this contract version. They may be introduced in future versions with dedicated message and state definitions:

- Typing indicators (SDK already has `send_typing` / `TypingActionType` — UI binding deferred)
- User presence / online status (SDK already has `PresenceStatus` — UI binding deferred)
- Message reactions (SDK already has `UpsertMessageReactionInput` — UI binding deferred)
- Message editing
- File / media attachments
- Thread / reply semantics
- Non-text message rendering: v0.3 `MessageVm.body` is text-oriented. Rendering for non-text `message_type` values (images, files, system messages, revoke notices) is deferred to a future version.
